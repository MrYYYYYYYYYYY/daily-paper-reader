## 摘要
数据整理是大语言模型后训练流程中的关键环节，然而现有工具往往将数据摄入、去重、合成生成和质量过滤视为独立阶段。这种碎片化使得审计流程决策或理解单个样本被拒绝的原因变得困难。CuratorKIT 是一个开源 Python 库，在单个可配置的流程中覆盖了完整生命周期。该框架由六种源格式读取器和自动模式检测、用于凭据、PII 和有害内容的预生成数据卫生层、八个由 LLM 驱动的生成任务、三个互补的质量门（带有精确溯源的幻觉验证）、结构化的自适应恢复，以及五种兼容 TRL、Unsloth 和 AlignTune 的训练就绪导出格式组成。每个流程决策都记录在仅追加的逐样本溯源链中，被拒绝的样本会附带结构化的失败原因，而不会被静默丢弃。CuratorKIT 通过 LiteLLM 支持 100 多个 LLM 提供商，提供了 Python API 和 YAML 驱动的 CLI，专为需要可复现、可审计的大规模数据流程的实践者设计。

## Abstract
Data curation is a critical part of post-training pipelines for large language models, yet existing tools often treat ingestion, deduplication, synthetic generation, and quality filtering as separate stages. This fragmentation makes it difficult to audit pipeline decisions or understand why individual samples are rejected. CuratorKIT is an open-source Python library that covers this full lifecycle in a single configurable pipeline. The framework is composed of six source format readers and automatic schema detection, a pre-generation data hygiene layer for credentials, PII, and toxic content, eight LLM-powered generation tasks, three complementary quality gates with provenance-exact hallucination verification, structured adaptive recovery, and five training-ready export formats compatible with TRL, Unsloth, and AlignTune. Every pipeline decision is recorded in an append-only per-sample provenance chain, and rejected samples carry structured failure reasons rather than being silently discarded. CuratorKIT supports 100+ LLM providers through LiteLLM, exposes both a Python API and a YAML-driven CLI, and is designed for practitioners who need reproducible, auditable data pipelines at scale .

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：现有LLM后训练数据整理工具将数据摄入、去重、合成生成、质量过滤视为独立阶段，各工具间schema不兼容、缺乏共用审计追踪，导致难以审计管道决策、难以理解单个样本被拒绝的原因，且无法实现迭代改进。
- **研究背景**：Distilabel、AgentInstruct、DataTrove、NeMo Data Designer等框架分别覆盖部分环节，但未能提供从源到导出的统一生命周期。特别是缺乏溯源导致幻觉门控时后验检索可能引入错误，而被拒绝样本被当作噪声丢弃则错失诊断和修复机会。
- **整体含义**：CuratorKIT旨在提供**单一可配置管道**，覆盖完整生命周期，记录每个样本的**仅追加溯源链**，并为被拒绝样本保留结构化原因，从而实现可复现、可审计的大规模数据管道。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：构建一个由四种基本步骤（读取器、门控器、归一化器、导出器）组成的流水线架构，所有处理决策记录在`DataSample`的`provenance_chain`中，被拒绝样本以`RejectedSample`形式保留结构化原因。
- **关键技术细节**：
  - **统一数据样本（DataSample）**：Pydantic模型，含13个字段，通过`task_type`标签区分任务类型，无需多个schema。
  - **溯源与拒绝**：每个步骤向`provenance_chain`追加不可变记录，包含步骤名、版本、时间戳、配置哈希等。拒绝时返回`RejectedSample`，包含`rejection_reason`（如"hallucination_contract_failed:0.42"）和`rejecting_step`。
  - **配置与API**：单一`CuratorConfig`（>100个字段），无嵌套子对象。提供Python API（`Curator(config).run()`）和YAML驱动的CLI。
  - **摄入与归一化**：支持6种格式读取器（JSONL、JSON、CSV、Parquet、HuggingFace Hub、PDF），自动格式检测（>60个列名变体映射，三级检测流程），`SchemaGate`强制字段/长度/编码要求，`TextCleaner`（5种变换），三种去重（精确、MinHash、基于嵌入的跨运行去重），分层采样与截断，五种导出格式（Alpaca、ShareGPT、DPO、GRPO、PPO、Corpus）。
  - **数据卫生层**：按成本递增顺序执行：`SecretsGate`（detect-secrets + 熵分析）、`PIIPseudonymizer`（Presidio + spaCy + Faker）、`ToxicityGate`（Detoxify分类器 + 可选LLM法官两阶段）。
  - **合成数据生成**：8个生成任务（QA、偏好、GRPO、多轮、演化指令、思维链、对抗性QA、不良样本注入），均继承`BaseGenerationTask`，通过`BaseLLM`接口（LiteLLMBackend / OllamaBackend）调用LLM，支持异步并发。
  - **质量过滤与自适应恢复**：
    - **幻觉门**：LLM法官根据源块对回答进行四维度评分（事实准确性、源接地、非矛盾、无参数漂移），产生`grounding_score` [0,1]，阈值默认0.7。溯源精确：使用生成时存储的源文本`sample.input`。
    - **奖励门**：按UltraFeedback维度（帮助性、诚实性、指令遵循等）评分，DPO对还需检查选择/拒绝分数对比。
    - **多样性门**：基于嵌入余弦相似度去重（默认0.92）。
    - **故障分类**：9种模式（6种可恢复），如`THRESHOLD_MARGINAL`、`GENERATOR_TEMPERATURE`、`INSTRUCTION_QUALITY`等。
    - **诊断探针**：对幻觉门拒绝的样本，根据接地分数路由到温度扫描或严格接地提示序列，最多5次LLM调用，首个通过的样本重新集成。
    - **奖励细化**：对奖励门拒绝的样本，读取最低分维度，进行单次定向重写，再重新评估。
  - **管道诊断**：`PipelineDiagnostics`收集每次运行中被诊断的拒绝，输出`diagnostic_summary.json`。

## 6. 论文的主要结论与发现

1. **门互补性**：幻觉门和奖励门捕获**几乎互斥**的失败样本（Jaccard重叠0.28~0.32），两者缺一不可。
2. **下游微调质量提升**：两个门过滤后，三个生成器规模的ROUGE-L、BERTScore-F1、Faithfulness均严格正提升，小模型（1.7B）提升最大（ROUGE-L +3.1%）。
3. **自适应恢复优势**：相比硬过滤，自适应恢复在1.7B/4B/8B生成器上分别多接受+23.6%/+48.4%/+53.1%样本；相比朴素重试，恢复率更高（32.5% vs 24.9%），且对四种幻觉类型的注入召回全面更高（总体67.0% vs 53.5%），特别是instruction_quality类型差距达23个百分点。
4. **数据卫生有效性**：SecretsGate在184个污染样本中**全部拦截**，无任何样本到达后续组件，证明成本排序的卫生层设计有效。

## 7. 优点

- **统一可审计**：单一框架覆盖完整生命周期，每个样本携带溯源链，拒绝样本附带结构化原因。
- **精确幻觉验证**：源文本保存在样本字段中，消除后验检索误差（RAG误差）。
- **自适应恢复**：根据故障类型定向修复（温度扫描、严格接地提示等），而非盲目重试，显著提升回收率。
- **灵活易用**：支持Python API、YAML CLI、100+LLM提供商（LiteLLM），与TRL、Unsloth、AlignTune兼容。
- **开源**：代码公开，便于社区扩展。
- **模块化设计**：通过抽象基类（BaseReader, BaseGate, BaseGenerationTask）可轻松添加新组件。

## 8. 不足与局限

1. **诊断探针延迟**：每个拒绝样本最多5次顺序LLM调用，虽可批量并行，但单样本延迟仍然较高。
2. **恢复适用范围有限**：当前仅支持基于源文本的生成任务（需要`sample.input`字段），指令只有提示（无源）的样本无法进入恢复流程。
3. **奖励细化仅一次迭代**：只进行一次重写和重新评估，若失败即永久拒绝，未实现循环优化。
4. **PII假名化默认英语**：使用Presidio英语模型，非英语语料需手动配置spaCy模型且准确性可能降低。
5. **不支持多模态**：仅处理文本。
6. **故障分类泛化性**：9种故障模式基于Qwen3和语料型QA任务开发，转移到其他任务/模型族需要额外实证。
7. **纵向对比不足**：未与其他成熟框架（如Distilabel）在相同条件下进行端到端流水线比较。
